# CLAUDE MD

## 1. 역할
- 이 문서는 개발 가이드와 구현 정책만 담는다.
- 기능 정의는 `docs/FEATURE_SPEC.md`, 화면 구조는 `docs/IA_SPEC.md`를 기준으로 한다.
- 현재 코드가 1차 기준이고, 문서는 그 구현을 설명하는 보조 기준이다.

## 2. 시스템 구조

| 구분 | 책임 |
|---|---|
| `app/*` | 라우트와 페이지 진입점 |
| `features/*` | 도메인별 화면, 상태, 액션 |
| `components/*` | 공통 UI 컴포넌트 |
| `api/*` | 현재는 mock 데이터/로직 레이어 |
| `types/*` | 도메인 계약과 상태값 |

- 현재 구조는 Next.js App Router + TypeScript 기반이다.
- 백엔드 실연결은 아직 없고, `api/*`가 사실상 계약 레이어다.
- 문서와 코드가 다를 때는 실제 구현을 우선한다.

## 3. 레이아웃 원칙

| 원칙 | 기준 |
|---|---|
| 공통 쉘 | 좌측 LNB + 상단 헤더 + 본문 |
| 상세 화면 | 목록과 상세를 분리한 2열 구조 우선 |
| 편집 화면 | 목록과 분리된 모달 사용 |
| 공통 리스트 패턴 | `ListPanel` 사용 |
| 상세 패턴 | 헤더 + 스크롤 영역 + 액션 영역 |

### `ListPanel` 구조
- `header`: 제목, 설명, 우측 액션
- `toolbar`: 검색, 필터, 보조 액션
- `body`: 실제 리스트/테이블 영역
- `footer`: 페이지네이션 또는 보조 정보

### 스크롤 정책
- LNB는 고정한다.
- 본문은 세로 스크롤을 사용한다.
- 리스트가 긴 화면은 `ListPanel` 내부 스크롤을 사용한다.
- 상세 패널은 리스트와 독립적으로 스크롤한다.
- 페이지 이동 시 전체 컨텐츠 높이는 흔들리지 않아야 한다.

## 4. 상태 관리 기준

| 상태 | UI 기준 |
|---|---|
| `loading` | 버튼 비활성화, 로딩 문구/스켈레톤 노출 |
| `empty` | 빈 상태 문구와 대체 안내 노출 |
| `error` | 상단 오류 메시지 또는 폼 하단 오류 노출 |
| `success` | 상단 성공 메시지 또는 결과 영역 갱신 |
| `disabled` | 권한 없음, 입력 미완료, 중복 제출 방지 |

- 로딩 상태에서는 중복 요청을 차단한다.
- 빈 상태는 오류가 아니다.
- 오류는 사용자 입력을 유지한 채 노출한다.

## 5. 메시지 정책

| 유형 | 노출 위치 | 규칙 |
|---|---|---|
| 성공 | 페이지 상단 메시지 | 2.5~3초 내 자동 소멸 |
| 실패 | 폼 하단 또는 모달 내부 | 입력값 유지 |
| 권한 없음 | 메뉴 비노출 또는 버튼 비활성화 | 가능한 경우만 안내문 노출 |
| 중복 | 제출 직후 즉시 차단 | 중복 사유를 명시 |
| 빈 데이터 | 리스트/상세 영역 | 오류와 구분된 빈 상태 문구 |

## 6. 권한 정책

| 메뉴 | MASTER | OPERATOR |
|---|---:|---:|
| 대시보드 | O | O |
| 콘텐츠 관리 | O | X |
| 캐시 Q&A | O | X |
| 지식 기반 조회 | O | O |
| 피드백 | O | O |
| 계정/권한 관리 | O | X |

- 메뉴는 역할 기반으로 숨긴다.
- 라우트가 노출되더라도 액션은 역할 조건에 따라 막는다.
- 자기 자신 계정은 변경/비활성화 대상에서 제외한다.

## 7. 페이지네이션 기준

- 현재 공통 페이지네이션 컴포넌트는 이전/다음 + 숫자 버튼 + 말줄임표를 지원한다.
- 전체 페이지가 7 이하일 때는 전부 노출한다.
- 7 초과일 때는 `1 ... current-1 current current+1 ... last` 패턴을 사용한다.
- 선택된 페이지는 항상 1~total 범위로 보정한다.
- 현재 페이지네이션이 적용된 주요 화면은 캐시 Q&A다.

## 8. 정렬 기준

- 기본 정렬은 최신 생성일 내림차순이다.
- 현재 구현에는 다중 정렬이 없다.
- 검색 결과는 관련도 우선 정렬 후 생성일로 보정한다.
- 정렬 기준은 화면별로 다르더라도 명시적으로 선언한다.

## 9. 검색 / 필터 기준

| 구분 | 기준 |
|---|---|
| 검색 | 화면 내 클라이언트 검색 우선 |
| 검색 방식 | 부분 일치 기반, 일부 화면은 유사도 정렬 포함 |
| 필터 | enum 값은 정확히 일치해야 한다 |
| 입력 전처리 | 공백 제거 + 소문자화 + 필요 시 특수문자 제거 |
| 결과 없음 | 빈 상태를 보여주고 오류로 처리하지 않는다 |

- 캐시 Q&A는 정규화 후 부분 일치와 유사도 점수를 함께 사용한다.
- 콘텐츠 관리는 문서명과 경로를 대상으로 부분 검색한다.
- 피드백은 반응 필터만 제공한다.
- 계정 추가 후보 검색은 이름, 아이디, 단지 코드 부분 일치다.

## 10. 로그 정책

| 범주 | 예시 |
|---|---|
| 인증 | 로그인, OTP, 로그아웃 |
| 관리자 작업 | 생성, 수정, 비활성화, 복구, 삭제 |
| 콘텐츠 작업 | 업로드, 수정, 다운로드, 삭제 |
| Q&A 작업 | 생성, 수정, 활성/비활성 전환, 삭제 |
| 피드백 조회 | 상세 열람 |
| 시스템 | 로딩, 빈 상태, 오류 |
| 검증 오류 | 필수값 누락, 중복, 권한 오류 |

- UI에 보이는 이력은 최신순으로 보여준다.
- 보관 기간과 저장 위치는 현재 구현에 없다.
- 보관 정책은 `OPEN_QUESTIONS`로 올린다.

## 11. API 필요 범위

- 현재는 실제 API가 아니라 mock 함수와 타입이 계약 역할을 한다.
- 상세 엔드포인트, 인증 헤더, 에러 바디는 확정되지 않았다.
- 백엔드 개발은 아래 수준으로 맞춘다.
  - 요청 파라미터 이름
  - 응답 객체의 필드
  - 상태값 enum
  - 오류 코드 규격

### 권장 오류 코드

| 코드 | 의미 |
|---|---|
| `E400_VALIDATION` | 입력값 누락/형식 오류 |
| `E401_UNAUTHORIZED` | 인증 안 됨 |
| `E403_FORBIDDEN` | 권한 없음 |
| `E404_NOT_FOUND` | 대상 없음 |
| `E409_CONFLICT` | 중복/충돌 |
| `E500_INTERNAL` | 예기치 않은 서버 오류 |

## 12. 성공 / 실패 후 UX 흐름

| 상황 | UX |
|---|---|
| 성공 | 목록/상세 갱신 후 성공 메시지 노출 |
| 실패 | 기존 입력 유지 + 오류 메시지 표시 |
| 검증 실패 | 모달을 닫지 않고 필드 교정 유도 |
| 권한 없음 | 아예 노출하지 않거나 비활성화 |
| 빈 결과 | 빈 상태 문구로 전환 |

## 13. 개발 규칙

- UI 로직과 도메인 로직을 분리한다.
- 상태값은 `types/*`를 기준으로 통일한다.
- 공통 패턴은 새로 만들기보다 기존 컴포넌트를 재사용한다.
- 추정이 필요하면 구현하지 말고 `OPEN_QUESTIONS`로 올린다.
